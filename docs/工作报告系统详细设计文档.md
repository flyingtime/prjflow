# 工作报告系统详细设计文档

## 1. 概述

工作报告系统用于管理员工的日报和周报，支持工作内容自动汇总、多审批人审批流程、审批记录管理。系统支持日报和周报的创建、更新、删除、提交、审批等操作，并自动从资源分配记录中汇总工作内容。

## 2. 功能特性

### 2.1 日报管理

- **日报CRUD**：创建、查看、更新、删除日报
- **日期唯一性**：同一用户在同一日期只能有一个日报
- **工作内容自动汇总**：自动从资源分配记录中汇总工作内容（需求、任务、Bug）
- **状态管理**：支持草稿（draft）、已提交（submitted）、已审批（approved）、已拒绝（rejected）状态
- **审批流程**：支持多审批人审批，每个审批人可以独立审批
- **审批记录**：记录每个审批人的审批状态和批注

### 2.2 周报管理

- **周报CRUD**：创建、查看、更新、删除周报
- **周范围管理**：支持设置周开始日期和结束日期
- **工作总结**：支持工作总结（Markdown格式）
- **下周计划**：支持下周计划（Markdown格式）
- **工作内容自动汇总**：自动从资源分配记录中汇总工作内容
- **状态管理**：支持草稿、已提交、已审批、已拒绝状态
- **审批流程**：支持多审批人审批，每个审批人可以独立审批
- **审批记录**：记录每个审批人的审批状态和批注

### 2.3 工作内容汇总

- **自动汇总**：从资源分配记录中自动汇总工作内容
- **工时统计**：统计总工时、按需求/任务/Bug分类统计工时
- **Markdown格式**：生成Markdown格式的工作内容摘要
- **数据来源**：
  - 资源分配记录（关联任务、Bug、需求）
  - 用户创建的Bug（即使没有资源分配记录）
  - 用户完成的任务（即使没有资源分配记录）

### 2.4 审批流程

- **多审批人**：支持设置多个审批人
- **独立审批**：每个审批人可以独立审批
- **审批状态**：待审批（pending）、已通过（approved）、已拒绝（rejected）
- **批注功能**：审批人可以添加批注
- **自动状态更新**：当所有审批人都已审批时，自动更新报告状态
- **拒绝处理**：如果有任何审批人拒绝，报告状态为已拒绝

### 2.5 报告筛选

- **我的报告**：查看自己创建的报告
- **待审批报告**：查看需要自己审批的报告
- **状态筛选**：按状态筛选报告
- **日期范围筛选**：按日期范围筛选报告
- **用户筛选**：管理员可以按用户筛选报告

## 3. 数据模型

### 3.1 日报表（daily_reports）

```go
type DailyReport struct {
    ID        uint      `gorm:"primarykey" json:"id"`
    CreatedAt time.Time `json:"created_at"`
    UpdatedAt time.Time `json:"updated_at"`

    Date        time.Time `gorm:"type:date;not null;uniqueIndex:idx_user_date" json:"date"` // 日期
    Content     string    `gorm:"type:text" json:"content"`                                  // 工作内容（Markdown）
    Status      string    `gorm:"size:20;default:'draft'" json:"status"`                    // 状态：draft, submitted, approved, rejected

    UserID uint `gorm:"index;not null;uniqueIndex:idx_user_date" json:"user_id"`
    User   User `gorm:"foreignKey:UserID" json:"user,omitempty"`

    // 审批人多对多关联
    Approvers []User `gorm:"many2many:daily_report_approvers;" json:"approvers,omitempty"`

    // 审批记录
    ApprovalRecords []DailyReportApproval `gorm:"foreignKey:DailyReportID" json:"approval_records,omitempty"`
}
```

**字段说明：**

- `Date`：日期，必填，与UserID组成唯一索引（同一用户同一日期只能有一个日报）
- `Content`：工作内容，Markdown格式，支持自动汇总
- `Status`：状态，默认"draft"，可选值：draft（草稿）、submitted（已提交）、approved（已审批）、rejected（已拒绝）
- `UserID`：用户ID，必填，关联用户表
- `Approvers`：审批人列表，多对多关联
- `ApprovalRecords`：审批记录列表

**唯一性约束：**

- 同一用户在同一日期只能有一个日报（通过`uniqueIndex:idx_user_date`保证）

### 3.3 日报审批记录表（daily_report_approvals）

```go
type DailyReportApproval struct {
    ID            uint      `gorm:"primarykey" json:"id"`
    CreatedAt     time.Time `json:"created_at"`
    UpdatedAt     time.Time `json:"updated_at"`

    DailyReportID uint         `gorm:"index;not null" json:"daily_report_id"`
    DailyReport   DailyReport  `gorm:"foreignKey:DailyReportID" json:"daily_report,omitempty"`

    ApproverID uint `gorm:"index;not null" json:"approver_id"`
    Approver   User `gorm:"foreignKey:ApproverID" json:"approver,omitempty"`

    Status  string `gorm:"size:20;default:'pending'" json:"status"` // 审批状态：pending, approved, rejected
    Comment string `gorm:"type:text" json:"comment"`                // 批注
}
```

**字段说明：**

- `DailyReportID`：日报ID，必填，关联日报表
- `ApproverID`：审批人用户ID，必填，关联用户表
- `Status`：审批状态，默认"pending"，可选值：pending（待审批）、approved（已通过）、rejected（已拒绝）
- `Comment`：批注，可选，审批人的批注内容

## 4. API设计

### 4.1 工作内容汇总API

#### 4.1.1 获取工作内容汇总

**接口：** `GET /api/reports/work-summary`

**权限：** 需要登录

**查询参数：**

- `start_date`：开始日期（YYYY-MM-DD），必填
- `end_date`：结束日期（YYYY-MM-DD），必填

**响应示例：**

```json
{
  "code": 200,
  "message": "success",
  "data": {
    "content": "## 需求\n\n- **需求1** (项目: 项目1) - 8.00小时\n\n**需求总工时**: 8.00小时\n\n## 任务\n\n- **任务1** (项目: 项目1) - 4.00小时\n\n**任务总工时**: 4.00小时\n\n## Bug\n\n- **Bug1** (项目: 项目1) - 2.00小时\n\n**Bug总工时**: 2.00小时\n\n**总工时**: 14.00小时",
    "hours": 14.0
  }
}
```

**汇总逻辑：**

1. 从资源分配记录中查询用户在指定日期范围内的资源分配
2. 按需求、任务、Bug分类汇总
3. 查询用户在指定日期范围内创建的Bug（即使没有资源分配记录）
4. 查询用户在指定日期范围内完成的任务（即使没有资源分配记录）
5. 生成Markdown格式的工作内容摘要

#### 4.2.7 审批日报

**接口：** `POST /api/daily-reports/:id/approve`

**权限：** 需要登录（必须是审批人）

**请求体：**

```json
{
  "status": "approved",
  "comment": "审批通过"
}
```

**字段说明：**

- `status`：审批状态，必填，必须是"approved"或"rejected"
- `comment`：批注，可选

**审批逻辑：**

1. 检查当前用户是否是审批人
2. 查找或创建审批记录
3. 更新审批记录状态和批注
4. 检查是否所有审批人都已审批
5. 如果所有审批人都已审批：
   - 如果有任何审批人拒绝，报告状态为"rejected"
   - 否则，报告状态为"approved"

## 5. 业务逻辑

### 5.1 工作内容自动汇总

工作内容自动汇总逻辑：

1. **查询资源分配记录**：
   - 从`resource_allocations`表查询用户在指定日期范围内的资源分配
   - 通过`resources`表关联`user_id`
   - 日期范围：`date >= startDate AND date < endDate+1`

2. **查询创建的Bug**：
   - 查询用户在指定日期范围内创建的Bug（即使没有资源分配记录）
   - 日期范围：`created_at >= startDate AND created_at < endDate+1`

3. **查询完成的任务**：
   - 查询用户在指定日期范围内完成的任务（即使没有资源分配记录）
   - 条件：`assignee_id = userID AND status = 'done' AND updated_at >= startDate AND updated_at < endDate+1`

4. **去重处理**：
   - 避免与资源分配中的Bug和任务重复

5. **生成Markdown摘要**：
   - 按需求、任务、Bug分类汇总
   - 显示项目名称、工时
   - 计算总工时

### 5.2 日报创建规则

- 同一用户在同一日期只能有一个日报
- 如果用户未提供`content`，系统自动汇总工作内容
- 创建日报时，如果指定了审批人，为每个审批人创建待审批记录

### 5.3 周报创建规则

- 周开始日期不能晚于周结束日期
- 如果用户未提供`summary`，系统自动汇总工作内容（从week_start到week_end）
- 创建周报时，如果指定了审批人，为每个审批人创建待审批记录

### 5.4 审批流程

审批流程规则：

1. **设置审批人**：
   - 创建或更新报告时，可以设置多个审批人
   - 为每个审批人创建待审批记录（status="pending"）

2. **审批操作**：
   - 只有审批人可以审批报告
   - 审批状态：approved（通过）或rejected（拒绝）
   - 可以添加批注

3. **自动状态更新**：
   - 当所有审批人都已审批时（没有pending状态的审批记录），自动更新报告状态
   - 如果有任何审批人拒绝，报告状态为"rejected"
   - 否则，报告状态为"approved"

4. **更新审批人**：
   - 更新报告时，如果更新了审批人列表，删除旧的审批记录，创建新的待审批记录

### 5.5 权限控制

- **查看权限**：
  - 用户可以查看自己创建的报告
  - 用户可以查看需要自己审批的报告
  - 管理员可以查看所有报告

- **更新权限**：
  - 用户只能更新自己创建的报告
  - 管理员可以更新所有报告

- **删除权限**：
  - 用户只能删除自己创建的报告
  - 管理员可以删除所有报告

- **审批权限**：
  - 只有审批人可以审批报告
  - 管理员可以审批所有报告

## 8. 注意事项

1. **日期唯一性**：同一用户在同一日期只能有一个日报，通过唯一索引保证
2. **自动汇总**：工作内容自动汇总从资源分配记录、创建的Bug、完成的任务中汇总
3. **审批流程**：支持多审批人，每个审批人独立审批，所有审批人都已审批时自动更新报告状态
4. **状态管理**：报告状态包括draft、submitted、approved、rejected，approved和rejected由系统自动设置
5. **权限控制**：确保用户只能访问和操作有权限的报告
6. **Markdown支持**：工作内容使用Markdown格式，前端需要使用Markdown渲染器
7. **日期格式**：日期使用"YYYY-MM-DD"格式，前端使用日期选择器
8. **审批记录**：每次设置审批人时，删除旧的审批记录，创建新的待审批记录
9. **数据完整性**：删除报告时，同时删除关联的审批记录和审批人关联
